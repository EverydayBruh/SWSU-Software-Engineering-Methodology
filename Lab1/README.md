# 1. Цель практической работы

Целью практической работы является приобретение знаний, умений и навыков для использования паттерна проектирования «Абстрактная фабрика» в проектировании информационно-вычислительных систем.

---

# 2. Основные понятия

При реализации проектов по разработке программных систем и моделированию бизнес-процессов встречаются ситуации, когда решение проблем в различных проектах имеют сходные структурные черты. Попытки выявить похожие схемы или структуры в рамках объектно-ориентированного анализа и проектирования привели к появлению понятия паттерна, которое из абстрактной категории превратилось в непременный атрибут современных CASE-средств.

Паттерны различаются степенью детализации и уровнем абстракции. Предлагается следующая общая классификация паттернов по категориям их применения:

* архитектурные паттерны,
* паттерны проектирования,
* паттерны анализа,
* паттерны тестирования,
* паттерны реализации.

**Архитектурные паттерны (Architectural patterns)** — множество предварительно определенных подсистем со спецификацией их ответственности, правил и базовых принципов установления отношений между ними. Они предназначены для спецификации фундаментальных схем структуризации программных систем. Наиболее известные паттерны этой категории — паттерны GRASP (General Responsibility Assignment Software Pattern). Эти паттерны относятся к уровню системы и подсистем, но не к уровню классов. Паттерны систематизировал и описал К. Ларман.

**Паттерны проектирования (Design patterns)** — специальные схемы для уточнения структуры подсистем или компонентов программной системы и отношений между ними.
Наиболее известные — паттерны GoF (Gang of Four), систематизированные Э. Гаммой, Р. Хелмом, Р. Джонсоном и Дж. Влиссидесом (всего 23 паттерна). Они не зависят от языка реализации, но реализация зависит от области применения.

**Паттерны анализа (Analysis patterns)** — схемы для представления общей организации процесса моделирования.
Они описываются в терминах предметной области. Наиболее известные — паттерны бизнес-моделирования ARIS (Architecture of Integrated Information Systems).

**Паттерны тестирования (Test patterns)** — схемы представления общей организации процесса тестирования программных систем.
К этой категории относятся, например, тестирование черного и белого ящиков. Их систематизировал М. Гранд. Некоторые реализованы в IBM Test Studio.

**Паттерны реализации (Implementation patterns)** — совокупность компонентов и элементов реализации, используемых при написании программного кода.
Они включают паттерны организации, оптимизации и устойчивости кода, а также разработки GUI. Описаны в работах М. Гранда, К. Бека, Дж. Тидвелла и др.

**Шаблон проектирования** (design pattern) — повторяемая архитектурная конструкция, представляющая собой решение проблемы проектирования в рамках часто возникающего контекста.

Порождающие паттерны (**Creational**) — абстрагируют процесс порождения классов и объектов. Они делают систему независимой от способа создания и композиции объектов.

Основные порождающие паттерны:

* Абстрактная фабрика (Abstract Factory) — предоставляет интерфейс для создания объектов, конкретные классы которых неизвестны.
* Строитель (Builder) — отделяет конструирование объекта от представления.
* Фабричный метод (Factory Method) — определяет интерфейс для создания объекта, но оставляет подклассам решение, какой класс порождать.
* Прототип (Prototype) — создает объект через клонирование другого.
* Одиночка (Singleton) — гарантирует наличие только одного экземпляра класса.

«Абстрактная фабрика» является порождающим шаблоном проектирования, который предоставляет интерфейс для создания семейств взаимосвязанных или взаимозависимых объектов без указания их конкретных классов.

**Преимущества применения паттернов проектирования:**

* суммирование опыта экспертов и его доступность рядовым разработчикам;
* создание общего словаря между разработчиками;
* упрощение понимания архитектуры по используемым паттернам;
* упрощение реструктуризации систем;
* повышение гибкости, сопровождаемости и повторного использования кода.

**Проблемы, которые могут возникать:**

* потеря гибкости проектирования;
* излишнее усложнение системы;
* слепое следование шаблонам, вызывающее архитектурные и логические проблемы.
